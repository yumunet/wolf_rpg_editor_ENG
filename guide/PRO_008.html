【A feature to increase the player character's movement speed with a specific key (Dash feature)】


Characters and events each have a movement speed set for them.
This time we will create what is known as a dash feature, where holding the Sub-key changes the player character's movement speed.

[Goal]

　1. Use key input to get whether the Sub-key is pressed.
　
　2. If the Sub-key is pressed, change the player character's movement speed.
　
　3. When the Sub-key is released, revert the player character's movement speed.

　■ Related topics
　【I want to detect whether a specific key has been pressed】

【Steps】

―― [1] Create the Common Event ――

　Since we want this to work on all maps, we will use a Common Event.
　Click the Common Event settings to open the Common Event editor.
　
　Select an available Common Event from the Common Event list.
　Give it a name so it is easy to identify.
　Set the Trigger Condition to "Parallel Process (Always)".
　

―― [2] Accepting key input ――　

　Next, double-click the bottom button or the Event Command display area
　to bring up the Event Command Input Window.
　
　Select "9: Key Input" from the Command Window,
　set the assignment destination to "This Common Self: 0",
　key type to "Basic", direction key to "Do Not Accept",
　and check only "Sub-key" while unchecking everything else.
　
　When done, click the "OK button".
　

―― [3] Creating the Conditional Branch ――　

　Select "7: Conditional Branch (Variable)", then first uncheck "Link 3 Variables".
　
　The settings from top to bottom are:
　Click the var button and set "V1: equals 1" (or a different variable if this one is already in use)
　"V1: equals -1"
　"This Common Self0: equals 12"
　Then check "Create an 'Otherwise' case",
　creating a total of 4 branches.
　
　Then click the "OK button".
　

―― [4] Changing movement speed (while dashing) ――　

　Leave the top two branches empty for now.
　Click to select the 3rd branch below them.
　
　From the Event Commands select "H: Event Control".
　Click "Character Movement" in the upper-right area.
　
　Click the drop-down at the top-left and choose "Player Character".
　There are many buttons so it is hard to find, but look near the lower-center for the "Speed/Frequency Settings" section.
　Click the drop-down and select "4".
　※ The higher the value here, the faster the character moves.
　
　Once selected, click the "Set Movement Speed" button below it.
　Press "OK".
　
　This completes the part that makes the player character dash.
　

―― [5] Changing movement speed (walking) ――　

　Next, create the process to revert the movement speed.
　This time click to select the bottom, Otherwise branch.
　
　As before, click on the movement specification in the command display area to open it.
　From "Speed/Frequency Settings", this time select speed "3" and apply it.
　
　When done, press "OK".
　

―― [6] Process when the Common Event stops ――　

　Finally, create a process to revert the movement speed when this Common Event stops.
　Without this, if the Sub-key is held down when the Common Event stops,
　the dash state would continue indefinitely.
　
　Click to select the 2nd branch from the top, "V1: is -1 or less".
　
　As before, configure it to select and apply speed "3" from movement specification.
　(You can also copy [C key] and paste [V key] from the Otherwise branch)
　
　
　Next, open "3: Variable Operation".
　
　Click the [Var] button and enter "[V1:] = 1".

　The dash Common Event is now complete.
　Try a Test Play.

　If the character speeds up while the Sub-key is held and returns to normal when released, you succeeded!
　Great work.

[Side Note]

　This dash Common Event may overwrite other movement specifications during event presentations/effects,
　potentially causing unexpected behavior.
　
　In that case, use Variable Operation to set "V1=1" beforehand
　to stop the dash Common Event's operation, which should resolve it.
　(So it goes through the top branch)
　
　To re-enable it, set "V1=0".
　
　Also, if you want to ensure the speed is definitely reverted,
　setting "V1=-1" via Variable Operation will revert the speed before stopping the operation.
　
　In that case, depending on the event's processing order, a 1-frame wait may be needed.
　Keep that in mind.
