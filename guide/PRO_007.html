【I want to know how to use the Database】

　【Question】

　The Database can manage a large amount of information.
　The Database exists to manage information in one place, and it excels especially at
　retrieving multiple similar pieces of information at once, such as
　map names, map filenames, and map BGMs.
　Once you understand how to use it, calling all item names becomes
　effortless, so it is an important feature if you want to create an encyclopedia or similar system.

　This time we will introduce what you can do with the Database
　and how to operate it for that purpose.

　【Explanation】
【What is the Database?】

· The game world contains a great deal of information, such as
　the player character's graphic (the image filename that serves as their appearance)
　and level (value).
　The place where this information is stored is the Database.
· If you already understand the concept of variables, you might think,
　"But variables store information too."
　The key difference between the Database and variables is whether
　values can be managed as a group.
　It is theoretically possible to build everything with variables alone and no Database,
　but the effort required is completely different.
【How do you use the Database?】

· The Database has a three-level hierarchical structure,
　named from outermost to innermost: Type, Data, and Item.
　You don't need to memorize the order by force — if you keep working with the Database
　as needed during game development,
　you will naturally remember it.
　The actual usable values, such as player character names and levels, reside in
　the innermost layer, "Item".

· The Database excels at managing the contents of its three-level hierarchy as groups at each level,
　making it extremely useful when building part or all of a system.

· For example, consider displaying a list of enemy character names and numbers
　for an enemy character encyclopedia.
　The Type represents the group "enemy character info",
　the Data represents the group for each individual enemy character,
　and the Item holds the enemy character's name, attack power, etc.
　In this case, by fixing the type number to "Enemy Character" and the item number to "Name",
　and then varying only the data number,
　you can retrieve only the name of each different enemy character.
　If you are using the Basic System, create any event and enter
　　\udb[9:X:0]
　in a message display, replacing X with any integer such as 0 or 1,
　then launch it in Test Play. This command calls
　User Database [Type 9 (Enemy Individual Data): Data X: Item 0 (Enemy Name)].
　Specify the enemy character's ID for data number X. Be careful not to specify
　a number that doesn't exist, as it won't work correctly.
【Types and uses of Databases】

· So far we have discussed the internal structure of the Database, but
　the Database is not just one type — each has a different purpose.
　There are three types: User Database, Changeable Database, and System Database.
　Let's look at each use case in detail.

1. User Database
· The User Database is also called UDB (User Data Base) for short.
　It stores "information that cannot be changed during the game",
　such as enemy strength, item effects, and menu content.
　It is a Database for information that does not change during the game,
　and the game creator must configure all type contents.
　In principle it cannot be written to; it exists to retrieve the information stored in it.
· In the Basic System it contains skills, items, enemy character settings,
　menu structure, etc., but people who build their own systems
　often use a completely different structure.
　The User Database is for information that is unique to each game and
　cannot be rewritten during the game
　(e.g., information you don't want the player to change).

2. Changeable Database
· The Changeable Database is also called CDB (Changeable Data Base) for short.
　It stores "information that can be changed during the game".
　Of the three Databases, only the Changeable Database can be directly manipulated during the game.
　This is the Database for players to use, and the game creator must build it individually for each game.
· The difference from the User Database is that the values inside can be
　changed during the game, making it very convenient for saving character stats, etc.
　The initial values are set by the game creator,
　but the contents change depending on the player's actions.
· By default it contains the player character's level and stats,
　party members, money held, etc.
　During combat, enemy character info and player character info are copied here
　to manage HP changes and similar values.

3. System Database
· The System Database is also called SDB or SysDB (System Data Base) for short.
　It manages "systems assumed to be used in any game".
　It contains BGM, map settings outlines,
　values that WOLF RPG Editor itself constantly monitors, and more.
· This is a Database whose structure was decided by WOLF RPG Editor's creator (SmokingWOLF).
　It contains things that WOLF RPG Editor fills in automatically (such as mouse coordinates in SDB[6:71])
　and things that the game creator configures (such as BGM settings and map settings).
· This Database is also not intended to be written to during the game;
　like the User Database, it exists for reading information.
　In the sense that "values cannot be written to it", it may appear similar to the User Database at first glance,
　but an important difference is that this Database has a predetermined structure,
　whereas the User Database's structure can be designed and changed by the creator.
· Also, this is quite important so please be careful:
　the type content structure of this Database is necessary for
　the normal operation of WOLF RPG Editor.
　The numeric or string values inside may be changed, but
　the "type contents" themselves should not be changed.
　【Details】
【Reading information from the Database】

　All items stored in the three types of Database can of course be read.
　There are three methods of retrieval: using Event Command "4 DB Operation" to
　write it to a variable first and then display that variable (indirect method);
　using a large integer to call it directly; or using a special character (the \udb[X:Y:Z] described above) to
　call it directly.
【Large Integers】

· In the various value input fields throughout WOLF RPG Editor,
　entering a specific large integer can call up a Database value in some cases.
　A large integer is a number of 1,000,000 or more that looks like an ordinary integer on the surface,
　but is actually a number configured to call a special value.

· For example, User Database values start from 1 billion,
　in the format "10AABBBBCC Type A / Data B / Item C".
　Entering "1010002233" calls the value in Type 10, Data 0022, Item 33.
　With the default data, this would call "0" stored in
　Type 10 "┣ Enemy Action AI", Data 22 "Dragon",
　Item 33 "┣9 Target [Single Target]".
　Similarly, Changeable Database and System Database values can be retrieved
　using large integers.
　Changeable Database starts from 1.1 billion, System Database from 1.3 billion.

　A list showing roughly which number ranges correspond to which values
　can be accessed from the Help menu under "Variable Call Values / Special Character List".
【Special Characters】

　Large integers can only retrieve numeric values, not strings.
　To retrieve strings, use special characters rather than numbers.
　The string "\udb[X:Y:Z]" that has appeared several times in this article
　is the special character format for calling Database values.
　The "\udb" part specifies the Database name, so:
　User Database → \udb,
　Changeable Database → \cdb,
　System Database → \sdb.
　For X:Y:Z, specify the Type, Data, and Item numbers respectively.

　Special characters are not exclusively for retrieving strings;
　they can also be used to retrieve numeric values. However, you cannot use either
　special characters or large integers everywhere at any time.
　Use special characters in places where strings are entered (message display, item names, etc.),
　and use large integers in places where numbers are entered (Variable Operations, etc.).

　Like large integers, these can be referenced from "Variable Call Values / Special Character List".
【Writing to the Changeable Database】

　Information can be written to the Changeable Database.
　Writing can be done via Event Command "4 DB Operation",
　and also via "3 Variable Operation".
