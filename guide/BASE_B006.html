【Display enemy names and counts at the start of combat】

[Goal]
　Immediately after all enemies are displayed, check the count for each enemy type and show a message.
　Reference: Triggering an event during combat
　>> Example <<
　○○○ appeared x3!
　△△△ appeared x2!


【Sample】
　


















[1]
　Select 203: ○[Modifiable] Combat Start Process
　from the Common Event list.




















[2]
　When 203: ○[Modifiable] Combat Start Process is called,
　all displayed enemies already have their data stored in
　Changeable DB_10: × ┏ Combat Temp Status [Base],
　so we will read the necessary values from that DB.




















[3]
　First we will read enemy character numbers from the DB,
　but checking them one by one to see if they are the same enemy is tedious.
　Let's use a Loop process to make it easier.

　Preparing for the Loop:

　1. Select Variable Operation from the Event Command list
　2. Select Self (slightly above), then select this Common Self 10
　3. Select = (Assign)
　4. Input 9 + 0
　　　Make it one less than the enemy 1 status data shown in the [2] image.
　　　It will become 10 after being incremented by +1 at the start of the Loop, so this is fine.
　5. Left-click Input to add it.





















[4]
　Next, add the Loop process.
　Enter 7 (the maximum number of enemies) as the Loop count.

　1. Select Event Control from the Event Command list
　2. Check "Loop with Count" and input 7
　3. Add it on the line below the CSelf10 = 9 + 0 you entered earlier

　Referring to [ 3 ], perform a Variable Operation and
　　　add CSelf10 += 1 + 0 inside the Loop.
　　　CSelf10 serves to save the data position in
　　　Changeable DB_10: × ┏ Combat Temp Status [Base].

　The Loop preparation is now complete.




















[5]
　Read the necessary values from the DB
　and copy them to Common Self variables.

　1. Select DB Operation from the Event Command Input
　2. Select Changeable DB and Name, then select × ┏ Combat Temp Status [Base]
　3. Select Variable and choose CSelf10
　　　※ Each Loop iteration increments CSelf10 by +1,
　　　changing it from 10 → 11 → ... → 16.
　4. Select Name and choose Character Number [10000+ = Enemy]
　　　If no character is present, -1 is stored.
　　　For enemy characters: UDB Enemy Individual Data position + 10000.
　　　For allies: CDB Player Character Status data position.
　5. Select Assign to Variable →
　6. For Value or Destination, select = (Assign)
　7. Select Variable/String and choose CSelf11
　　　The enemy character number we are reading is a value,
　　　so it cannot be assigned to a string variable.
　　　Assign it to a non-string variable.
　8. Click Input and add it on the line after the CSelf10 += 1 + 0 from before.




















[6]
　Branch conditionally based on the read value.

　The Character Number [10000+ = Enemy] in × ┏ Combat Temp Status [Base] has:
　initial value -1, ally characters are 0–9999,
　and enemies are 10000–19999.
　So the branch condition is: 10000 or more AND 19999 or less.

　1. Select Condition (Variable) from the Event Command list
　2. Select the variable containing the enemy character number read earlier
　3. Input 10000
　4. Select or more
　5. Click Input and add it.

　Similarly, create a branch for 19999 or less inside the branch you just created.
　This ensures the process only runs when the character number is
　10000 or more AND 19999 or less.




















[7]
　Next, use a Changeable DB and similarly run a Loop process
　to check whether the same character number already exists.

　First, decide which Changeable DB to use.
　This time we will use 16: × Shop Temp Process,
　which is likely not used during combat
　and has all its data initialized before use.
　Of course, you can also use 19: All-Purpose Window Temp DB or
　create a dedicated DB in Changeable DB slot 20 or later.
　The created DB just needs 2 entries: enemy character number and appearance count.
　Set the initial value for the enemy character number entry to -1, and appearance count to 0.
　※ Data positions cannot use negative values (-1 or less),
　※ so setting -1 as the initial value prevents erroneous operations.

　Once you decide which DB to use, add a command to initialize all data in that DB
　on the line immediately above the very first Loop.
　If you don't think carefully about where to add it,
　all your hard work on the process might go to waste.




















[8]
　Now let's use the prepared DB to save information about the appearing enemies.

　1. Inside the "19999 or less" branch, add CSelf12 = -1 + 0
　　　CSelf12 is the variable for saving the data position in the prepared DB.
　2. Add a Loop with Count of 7
　　　Set to 7, the maximum number of enemies that can appear.
　3. Move inside the Loop in step 2 and add CSelf12 += 1 + 0
　　　This increments the data position by +1 each iteration.
　4. Using DB Operation,
　　　copy the Price of × Shop Temp Process at CSelf12 to CSelf13
　　　※ In × Shop Temp Process used here, when data is initialized,
　　　※ the Price entry for all data becomes the initial value of 0,
　　　※ so use that value and the character number to branch on the next line.
　5. Branch on whether CSelf13 equals 0 and
　　　whether CSelf13 equals CSelf11
　　　If the initial value is -1, set "same as -1"; if -5, set "same as -5".
　　　※ Since we are only checking enemies this time,
　　　※ only values of -1 or 10000–19999 will appear,
　　　※ so an initial value of 0 is fine.
　　　※ If you also need to search ally characters,
　　　※ the initial value must always be -1 or less,
　　　※ because 0 could match Player Character Status data position 0.
　6. Inside the branch where CSelf13 equals 0, use DB Operation:
　　　× Shop Temp Process at CSelf12: Price = CSelf11
　　　Also increment Quantity by += 1
　　　※ When it's 0 (initial value), the slot is empty,
　　　※ so register the enemy character number, and increment the appearance count entry by +1.
　　　※ Since the initial value of appearance count is 0, registration and counting as 1 happen simultaneously.
　7. Inside the branch where CSelf13 equals CSelf11, use DB Operation:
　　　× Shop Temp Process at CSelf12: Quantity += 1
　　　※ Here the enemy character number registered in the DB simply matched
　　　※ the enemy character number at the current position,
　　　※ so just increment the appearance count by +1.
　　　※ If neither matched, continue to the next Loop.




















[9]
　At the last line inside both branches,
　add a Break Loop command.
　Forgetting this will cause the process to malfunction.
　※ Enemy character numbers could be duplicated across other data positions.
　※ For example, if the enemy group is 2 Wolves and 1 Slime:
　※ Wolf appeared x2!
　※ Wolf appeared x2!
　※ (Same text repeating through line 7)

　
>>> Flow of the process so far <<<

　Retrieve the enemy character number from all 7 slots one by one.
　If the value is 10000 or more AND 19999 or less,
　use the prepared Changeable DB to save the enemy character number
　and count and save the appearance count by type.






















[10]
　Now we will finally display the names of the appearing enemies.
　Start adding from the line after exiting the very first Loop.




















[11]
　First, set values for the string variables to use,
　the variable for counting enemy types,
　and the variable for the DB data read position.
　Let's assign an empty string to CSelf5.
　1. Select String Operation from the Event Command list
　2. Select String and specify Common Self 5
　3. Select = (Assign)
　4. Leave the field empty — do not press Enter or add spaces
　　　Entering any character here would change what appears on the first line of the message.
　Press Input at the end and you are done.
　Similarly, assign an empty string to CSelf6.

　As for the purpose of the variables:
　String variables CSelf5 and CSelf6 are for displaying the enemy type names,
　counts, and the appearance message text.
　Use CSelf10 for the DB data read position.
　Use CSelf11 for counting the number of enemy types.
　Set CSelf10 = -1 + 0
　and CSelf11 = 0 + 0.




















[12]
　Add the following on the line after CSelf11 = 0 + 0.
　1. Set the Loop count to 7, the maximum number of enemies that can appear, and add it.
　2. Move inside the Loop and add 1 to CSelf10 (data position counter).
　　　This increments the data read position from 0 → 1 → 2 ... 6 each iteration.
　3. Assign (=) the value from the DB where enemy character numbers are saved to CSelf12
　　　※ Read from the DB you prepared here,
　　　※ not from × ┏ Combat Temp Status [Base].
　4. Branch on whether the enemy character number assigned to CSelf12 is not 0 vs. 0
　　　When it's not 0 (initial value of the prepared DB entry),
　　　an enemy character number (enemy individual data number + 10000) is stored.
　　　When it matches the initial value, everything beyond is also the initial value,
　　　so create a branch to break out of the Loop.




















[13]
　Move into the branch for when the enemy character number is set:
　　　The branch for when CSelf12 is not 0 (item initial value).

　1. Add 1 to CSelf11 (+=)
　　　CSelf11 is the variable saving the total number of enemy types.
　　　For an enemy group of Dog, Cat, Monkey, Cat, Monkey:
　　　the types are Dog, Cat, Monkey, so the total type count is 3.
　2. Divide CSelf12 by 10000 and get the remainder
　　　Since enemy character numbers are 10000–19999,
　　　this process obtains a value of 0–9999.
　3. Assign (=) the appearance count from the saved DB to CSelf13
　　　※ Read from the DB you prepared here,
　　　※ not from × ┏ Combat Temp Status [Base].
　4. Assign (=) the character name from UDB's 9: Enemy Individual Data to CSelf7
　　　CSelf12 contains the data position of the enemy individual data.
　　　Character names are strings so they can only go into string variables.
　5. Branch on whether CSelf11 is 4 or less vs. 7 or less
　　　When using the default font size,
　　　messages can only display up to 4 lines.
　　　If 4 or less, use CSelf5;
　　　if 5–7, use CSelf6.
　6. Build the enemy character name, appearance count and text string to display here.
　　　Concatenate the combined string into the string variable (+=).
　　　※ \cself[7] holds the character name, \cself[13] holds the appearance count.
　　　※ \n is the newline special character; forgetting it will result in:
　　　※ Dog appeared x1!Cat appeared x1!Monkey...
　　　※ running together on one line.




















[14]
　Explanation of the process from [11] to [13]:
　Loop 7 times (maximum enemy appearances),
　and during each Loop, retrieve the enemy character number registered in the DB.
　If the character number is not 0 (item initial value), it is registered,
　so the branch makes the Loop exit when the character number matches the initial value.

　If it was not 0, increment CSelf11 (enemy type count variable) by +1,
　and read the appearance count into CSelf13.

　Read the character name for the character number from the UDB enemy individual data.
　Branch on whether the enemy type count is 4 or less vs. 7 or less.
　　　※ In practice this is branches of 1–4 and 5–7.
　If 4 or less, concatenate (+=) the enemy character name and appearance count into CSelf5;
　if 7 or less, concatenate into CSelf6.

　Later, Show Message will display the enemy names etc. as a message,
　but with the default font size only 4 lines can be displayed,
　so from the 5th type onward it goes into CSelf6 and Show Message is called twice.
















[15]
　1. On the line after the end of the Loop,
　　　add a branch conditioned on CSelf11 (enemy type count).
　　　※ Put the "5 or more" branch first,
　　　※ then use "1 or more" as the next condition
　　　※ to split into "1–4" and "5 or more" branches.
　2. Inside the branch, add a Show Message.
　　　If there are 5 or more enemy types,
　　　display CSelf5 first and then also display CSelf6.
　　　Since the text is split 4 lines each into separate string variables,
　　　each string variable must be displayed separately.





















[16]
　Now let's trigger a combat and check what happened.

　Refer to ◆ I want to trigger a battle
　to trigger a combat against a suitable opponent.

　After all enemies are displayed,
　were the enemy character names and appearance counts displayed as a string message?

　If they were, you succeeded.

　>>> Possible reasons if it doesn't work <<<
　1. The DB data read position is wrong.
　2. The Loop count is wrong.
　3. The item initial value and the initial value used in the Conditional Branch don't match.
　4. The specified Common Self variable is wrong.
　5. You forgot to apply %=10000 to the character number.
　6. You forgot to add \n (newline character) at the end of the string variable.





















[17]
　Using special characters like
　【Character Property Changes】 and 【Text Display Control】 in your strings will expand your presentation/effect options.

　Also, adding some wait time at the end of the text display branch
　lets you adjust the interval before the Fight/Flee choices appear.

　Whether your process worked or not, good work!
